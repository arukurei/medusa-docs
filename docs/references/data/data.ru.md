# Data

Работа с данными в Medusa построена на **трехслойной архитектуре**. Понимание того, как внешние базы (`MedusaDB`, `ProgressDB`) взаимодействуют с внутренними реестрами Графа, критично для создания производительных и стабильных деревьев.
Система Medusa автоматически связывает визуальные ноды с их логическими данными через систему ID.

## Внешние Ресурсы (Контейнеры)

1.  **MedusaDB (Каталог):**
    *   **Зачем:** Хранит в памяти все `AtomData`, которые сейчас присутствуют на сцене.
    *   **Механика:** Использует **подсчет ссылок** (Reference Counting). Когда вы добавляете Атом на граф, он регистрирует свой `AtomData` в базе. Если вы удалите все Атомы с ID `&"fireball"`, ресурс автоматически удалится из `MedusaDB` для экономии памяти.
    *   **Связь:** `AtomData.id` является ключом для поиска.
2.  **ProgressDB (Состояние игрока):**
    *   **Зачем:** Хранит только факты: что куплено и какого уровня.
    *   **Механика:** Обычный словарь `{StringName ID : int Level}`. Это минимальный объем данных, который легко сериализовать в JSON или бинарный файл для сохранений.

!!! tip
    Так как `ProgressDB` — это обычный ресурс (`Resource`), вы можете использовать стандартные методы Godot.
    **Пример сохранения:**
    ```gdscript
    func save_game():
        ResourceSaver.save(progress_db, "user://save_progress.tres")
    ```
    **Пример загрузки:**
    ```gdscript
    func load_game():
        if FileAccess.file_exists("user://save_progress.tres"):
            progress_db = load("user://save_progress.tres")
            graph.sync_with_progress(progress_db) # Сразу обновляем дерево
    ```
    Совет: Используйте расширение `.tres` для отладки (текстовый формат) и `.res` для финальной игры (бинарный формат).*

## Внутренние Реестры Графа (Ядро)

Когда вы запускаете игру или добавляете ноду через `add_atom()`, Граф не просто отрисовывает её, а индексирует в нескольких внутренних словарях. Это позволяет Графу выполнять проверки (например, "найди всех соседей с тегом 'Fire'") мгновенно (O(1)), а не перебором всех нод (O(n)).

#### 1. Идентификационный реестр (`_id_registry`)

Связывает текстовый ID из `AtomData` с конкретными экземплярами нод `Atom` на сцене.
*   **Нюанс:** Один ID может принадлежать нескольким нодам (например, если у вас на экране два одинаковых итема). Граф вернет массив всех подходящих нод.

#### 2. Реестр тегов (`_tag_registry`)

Кэширует атомы по их тегам.
*   **Как это работает:** Если вы запрашиваете через логику "проверить все ноды с тегом `&"root"`", Граф не ищет их по сцене — он просто берет готовый список из этого словаря.

#### 3. Топологические карты (`_parents_map` и `_children_map`)

Это самая важная часть для деревьев навыков. Когда вы указываете `Connected Paths` (связи):

*   Граф анализирует направление связи.
*   Он строит карту "Предков" и "Потомков".
*   Метод `graph.has_unlocked_neighbor()` использует эти карты, чтобы мгновенно понять, можно ли разблокировать текущий атом на основе состояния его родителей.

## Процесс Синхронизации (Жизненный цикл)

Синхронизация — это момент, когда "Мертвое" дерево навыков оживает под воздействием данных игрока.

**Вызов `graph.sync_with_progress(progress_db)` запускает следующий цикл:**

1.  **Обновление Контекста:** Граф берет актуальное состояние из переданной `ProgressDB`.
2.  **Проход по логикам (Logics):** Для каждого Атома вызываются назначенные в Графе `GraphLogic`.
3.  **Применение Состояний:** 
    *   Логика спрашивает: "Есть ли ID этого атома в ProgressDB?". 
    *   Если да — `status = UNLOCKED`. 
    *   Если нет, логика проверяет соседей через `_parents_map`. 
    *   Если есть разблокированный сосед — `status = AVAILABLE`.
4.  **Визуальное обновление:** После смены `status`, Атом автоматически переключает свои **AnimationLibrary**, меняя иконку, цвет или запуская эффекты (например, свечение доступного навыка).

!!! note
    Если вы создаете дерево программно, вам не нужно вручную менять `status` у каждого Атома. Ваша задача — только **настроить связи** и **добавить ID в ProgressDB**. Метод `sync_with_progress()` сделает всю остальную работу, опираясь на правила в `graph_logics`.
    **Пример автоматизации:**
    ```gdscript
    func _on_buy_button_pressed(atom: Atom):
        progress_db.unlock(atom.data.id) # 1. Просто добавляем ID в сохранение
        graph.sync_with_progress(progress_db) # 2. Граф сам поймет, какие НОВЫЕ атомы теперь стали AVAILABLE
    ```
